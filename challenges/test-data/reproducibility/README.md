# Reproducibility Test Data

Test fixtures for the current Hermes reproducibility flow using the CSV comparison scorer in `containers/repro-scorer`.

This folder is meant to feel like a small but believable computational reproducibility bounty:
- public input data is given to solvers
- a reference output is used for scoring
- solvers submit a reproduced CSV artifact
- score reflects row-by-row agreement with optional numeric tolerance

## Current Runtime Contract

The reproducibility scorer currently expects:
- `ground_truth.csv`: the reference output artifact
- `submission.csv`: the solver's reproduced output
- identical column names between both files

The built-in preset is:
- `CSV Comparison`
- official container: `ghcr.io/hermes-science/repro-scorer:latest`

## Files

### Challenge posting inputs

| File | Use in UI | Purpose |
|------|-----------|---------|
| `input_dataset.csv` | Input dataset | Public source data solvers work from |
| `expected_output.csv` | Expected output | Reference artifact used for scoring |

### Solver submission fixtures

| File | Expected result | Purpose |
|------|-----------------|---------|
| `sample_submission.csv` | Valid, perfect score | Exact reproduction of expected output |
| `near_match_submission.csv` | Usually valid if tolerance allows | Small numeric drift within tolerance |
| `bad_submission_missing_column.csv` | Invalid | Missing a required output column |
| `bad_submission_extra_rows.csv` | Valid but penalized | Reproduced artifact has mismatched row count |

### Local scorer convenience

| File | Purpose |
|------|---------|
| `ground_truth.csv` | Direct local alias for `expected_output.csv` |

## Recommended Human UI Test Plan

## 1. Post a realistic reproducibility challenge

Use these values in `/post`:

### Section 1: Problem
- Title: `Reproduce normalized assay summary statistics from a published pipeline`
- Domain: `Longevity` or `Other`
- Description:
  `We provide a public input dataset and expect solvers to reproduce the normalized summary table generated by the reference pipeline.`
  `Submissions should be CSV artifacts with the same columns as the expected output.`
  `Scores are computed by row-by-row comparison against the reference artifact.`
- Tags: `reproducibility, csv, benchmark`

### Section 2: Data
- Input dataset: `input_dataset.csv`
- Expected output: `expected_output.csv`

### Section 3: Evaluation
- Scoring method: keep the default `CSV Comparison` preset
- Tolerance: `0.001`
- Scoring description:
  `Rows are compared positionally against the expected output, with numeric values accepted within tolerance 0.001.`

### Section 4: Reward & Execution
- Reward: `10`
- Winner selection: `Winner takes all`
- Deadline: `7 days`
- Dispute window: any dropdown value you want to test

## 2. Submit the exact reproduction

Use:
- `sample_submission.csv`

Expected current behavior:
- perfect score
- all rows matched
- no row-count penalty

## 3. Submit the near-match reproduction

Use:
- `near_match_submission.csv`

Why:
- validates the tolerance field in a realistic way
- useful for checking whether the challenge poster chose a tolerance that is too strict or too loose

Expected current behavior:
- with tolerance `0.001`, this should still score very highly or perfectly
- with tighter tolerance, some rows may stop matching

## 4. Submit malformed or penalized outputs

### Missing column
Use:
- `bad_submission_missing_column.csv`

Expected current behavior:
- rejected because the submission is missing a required column from the expected artifact

### Extra rows
Use:
- `bad_submission_extra_rows.csv`

Expected current behavior:
- accepted but penalized for mismatched row count
- useful for understanding how the current scorer handles over-produced output artifacts

## Why This Folder Is Realistic

A lot of reproducibility bounties are not arbitrary notebooks. They are effectively:
- `here is the input data`
- `here is the reference artifact`
- `reproduce this exact result within a documented tolerance`

That is exactly what this folder is modeling.
